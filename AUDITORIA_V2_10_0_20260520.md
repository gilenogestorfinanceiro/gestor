## TL;DR (leitura honesta)

A função **não está tão ruim quanto o ROADMAP_v2.10.0 sugere**, mas tem **5 bugs latentes reais** e uma **assimetria estrutural** que vai voltar a morder em qualquer expansão de escopo (foi o que aconteceu na v2.9.50).

- **Conversão objeto→array em memória:** **NÃO existe mais.** Achado histórico do DIARIO está superado.
- **Limitação de 3 meses:** **EXISTE e é estrutural**, não cosmética. Não é só "olhar mais meses" — a função tem uma assimetria entre **limpeza global** e **recriação local** que torna qualquer expansão de range perigosa sem refactor.
- **Risco real de corrupção em produção hoje:** **médio-baixo, mas não-zero.** Há 2 cenários plausíveis que perdem dados de usuário silenciosamente (detalhados em §2).
- **Refactor proposto (upsert por chave) é o caminho certo**, mas a **chave correta não é `cartão+mês+ano`** sozinha — precisa incluir o `cpList.length === 0 || allPaid` como condição de existência, e o `done`/`userEdited` como axes ortogonais.
- **Estimativa do ROADMAP (3 sessões de 1-2 semanas)**: realista. Não tem como fazer em 1 sessão honestamente.

**Achado §1.2.a (importante):** No call site #1 (`loadFromCloud`), **não há `save()` depois**. A função muta `D` em memória após carregar do Firestore, mas o estado limpo não é persistido até a próxima ação do usuário. Consequência:
- Se a base na nuvem tem lixo legado (eventos `venc_*`, faturas órfãs), a tela mostra estado limpo, mas a nuvem continua suja.

### 1.4 Caminho de execução (uma passagem)

```
[1] Garante D.ag, D.tx como arrays
[2] Computa janela: prev(cMonth-1), current(cMonth), next(cMonth+1)
[3] Snapshot _doneMap: { id → true } para todo evento de fatura com done=true
[4] LIMPEZA GLOBAL em D.ag (todos meses, qualquer ano):
    - preserva done=true
    - preserva userEdited=true
    - remove isFaturaEvent=true
    - remove ids "venc_*" ou "agfat_*"
    - remove títulos exatamente "Fatura <card>" e qualquer título iniciando por "Fatura "
[5] LIMPEZA GLOBAL em D.tx (todos meses):
    - remove pgtoFatura && st='agendado'
    - remove faturaAgendada && st='agendado'
[6] Loop meses × cartões:
    - Se não há cp na (m,y,card) → skip
    - Se todos cp.st==='paga' → skip
    - Push em D.tx: tx agendado com id determinístico agfat_tx_<card>_<m>_<y>
    - Push em D.ag: evento agendado com id determinístico agfat_<card>_<m>_<y>
    - Restaura done de _doneMap[agId] se existir
```

**Observação crítica:** A assimetria entre [4]/[5] (GLOBAL) e [6] (3 MESES) é o coração do problema. Está plantada na função desde a v2.9.45+.

## 2. Achados não-óbvios (bugs latentes reais)

### 🟥 Achado #1 — Fatura em aberto **fora da janela de 3 meses** é apagada e não recriada

**Onde:** filtros em [index.html:3889-3904](index.html:3889) vs loop em [index.html:3908](index.html:3908).

**Cenário concreto:**
1. Usuário em 20/05/2026 (cMonth=4, cYear=2026 — janela: Abr/Mai/Jun 2026).
2. Lança hoje uma **compra parcelada em 12x** num cartão (Mai/26 a Abr/27).
3. `sincronizarFaturasEmAberto` é chamado em [index.html:2403](index.html:2403).
4. Recriação só cria eventos para Abr/Mai/Jun 2026. Jul/2026 até Abr/2027 ficam **sem evento e sem tx agendado** — mas as `D.cp` existem.
5. Resultado: usuário não vê a fatura futura na agenda, não vê o agendamento no extrato.
6. **Pior:** quando usuário navegar para Jul/2026 (cMonth=6), a janela passa a ser Jun/Jul/Ago e os eventos de Jul são criados — mas Set/2026+ continuam invisíveis.
7. **Pior ainda:** se o usuário marcar o evento de Jul/2026 como `done=true` manualmente (sem pagar), e depois navegar de volta para Mai/2026, próximo sync **preserva** Jul (done=true) — mas isso só funciona se o evento existir. Se ele nunca foi criado porque a janela nunca cobriu Jul, nada é preservado.

**Por que ROADMAP_v2.10.0 trata isso como cosmético e está enganado:** o plano diz que a chave de upsert resolve. Resolve sim, **mas só se a função for chamada para todos os meses relevantes**. Hoje a função sequer **olha** Jul/Ago/Set/.../Abr+1 — a chave de upsert sem expansão de range não muda nada. O refactor precisa de **duas coisas**, não uma: (a) chave determinística + (b) range derivado de `D.cp` (todos os meses onde há cp aberto), não de `cMonth±1`.

**Severidade:** alta. Parcelas longas (12x, 24x, 36x) são caso real: itens de educação, eletrônicos. Provavelmente alguns dos 35 usuários têm isso hoje.

### 🟧 Achado #2 — Eventos custom do usuário com título iniciando por "Fatura " são apagados silenciosamente

**Onde:** [index.html:3896](index.html:3896) — `if(e.title && e.title.trim().startsWith('Fatura ')) return false;`

**Cenário concreto:**
1. Usuário cria lembrete na agenda: "Fatura conta de luz Equatorial" (não é cartão — é lembrete).
2. `saveAgenda` em [index.html:4047](index.html:4047) faz `D.ag.push(ev)`. Evento criado **sem** `userEdited=true` (a flag só é setada no path de EDIÇÃO, [index.html:4027](index.html:4027), não no de CREATE).
3. `sincronizarFaturasEmAberto` **não é chamado** aqui porque `agType!=='lembrete'` é false (é lembrete). [index.html:4056](index.html:4056).
4. Usuário fica feliz, evento aparece na agenda.
5. **Algum tempo depois**, usuário cadastra uma compra no cartão → call site #2 dispara sincronizarFaturas.
6. Filtro [index.html:3896](index.html:3896) deleta o evento "Fatura conta de luz Equatorial" porque o título começa com "Fatura ".
7. Usuário perde o lembrete sem aviso.

**Mitigação parcial:** se o usuário marcar o evento como `done=true`, é preservado ([index.html:3890](index.html:3890)). Se editar, vira `userEdited=true`. Mas isso depende de comportamento do usuário, não da função.

**Severidade:** média. Bug latente, provavelmente nunca foi reportado porque "Fatura " como prefixo de lembrete é incomum, mas é totalmente plausível.

### 🟨 Achado #3 — Estado `done` perdido em eventos com ID legacy `venc_*`

**Onde:** [index.html:3881-3885](index.html:3881) (snapshot por `e.id`) vs [index.html:3942](index.html:3942) (restore por `agId` formato novo `agfat_*`).

**Cenário concreto:**
1. Base antiga (pré-v2.9.46) tem evento `id="venc_NubankRoxinho_4_2026"` com `done=true` e `title="Fatura Nubank Roxinho"`.
2. Snapshot salva: `_doneMap["venc_NubankRoxinho_4_2026"] = true`.
3. Limpeza apaga o evento (matches `e.id.startsWith('venc_')`).
4. Recriação gera novo `id="agfat_Nubank_Roxinho_4_2026"` (cartão sanitizado com `\s+` → `_`).
5. Restore tenta `_doneMap["agfat_Nubank_Roxinho_4_2026"]` → **undefined**. `done` fica `false`.
6. Evento ressurge como "pendente" mesmo tendo sido marcado pago.
7. UX: usuário vê fatura "não paga" mesmo após ter pago no histórico antigo.

**Mitigação real:** se `D.cp` daquela fatura tem `st='paga'`, a recriação **nem cria** o evento (linha 3913: `if(allPaid) return`). Então o impacto é só em casos de transição (cp aberto + done legacy). Provavelmente cenários raros, mas existem em bases antigas dos 35 usuários.

**Severidade:** baixa-média. Difícil acionar em produção atual (a maioria dos legacys já passou por re-sync), mas auditável.

### 🟧 Achado #5 — `done=true` em evento de fatura cria duplicata quando a fatura é reaberta

**Onde:** lógica de [reopenFatura](index.html:2959) vs [sincronizarFaturasEmAberto](index.html:3858).

**Cenário concreto:**
1. Fatura Abr/2026 paga. `D.cp.st='paga'`, evento `agfat_*_3_2026` com `done=true`, tx agendado removido em [index.html:2932](index.html:2932).
2. `reopenFatura` ([index.html:2959](index.html:2959)) reabre: muda `D.cp.st` para `'aberta'`, remove tx de pagamento, **mas NÃO toca no `done` do evento** (linha 2974 só chama sincronizar).
3. Sincronizar entra:
   - Snapshot: `_doneMap["agfat_*_3_2026"] = true` (porque done=true).
   - Limpeza: evento `done=true` é **preservado** (linha 3890), **não é removido**.
   - Recriação: como cp agora tem aberta, recria evento com mesmo `id="agfat_*_3_2026"` e `done=true` (do _doneMap).
   - **`D.ag.push` é chamado sem checar duplicata.** Resultado: 2 eventos com mesmo ID na agenda.
4. UX: usuário vê duas linhas idênticas de "Fatura X" na agenda do mesmo dia.

**Confirmação por leitura:** linha 3934 não dedupe, e linha 3890 preserva ANTES do push. **Bug confirmado.**

**Severidade:** média. Reopen é operação rara (usuário se arrependeu de pagar). Em 35 usuários × 7 ativos provavelmente nunca foi acionado, mas é determinístico.

**Mitigação possível em pré-refactor (1 linha):** trocar [index.html:2974](index.html:2974) por algo que primeiro faça `agEv.done=false` antes de chamar sincronizar. Mas isso é patch — refactor resolve estruturalmente.

## 3. Intenção declarada vs realidade

| O que o nome / contexto sugere | O que a função faz de verdade | Gap |
|---|---|---|
| "Sincronizar faturas em aberto" — operação idempotente que reconcilia estado derivado | Faz isso, mas com janela arbitrária de 3 meses centrada em `cMonth` global | ⚠️ Sincroniza apenas uma fatia, não o todo |
| Função "read-mostly" — só altera estado para refletir cp | Apaga D.ag/D.tx globalmente e recria localmente | ⚠️ Assimetria global-vs-local é fonte de bugs |
| Idempotente: rodar 2x não muda o resultado | É idempotente em condições normais — exceto em #5 (reopen) e quando cp tem mês fora da janela | ⚠️ Quase idempotente, com cantos vulneráveis |
| "Em aberto" = não-pagas | Cobre cp.st!=='paga', mas também afeta eventos com título "Fatura " mesmo não-faturas | ⚠️ Escopo de match maior que a intenção |
| Função interna que reconcilia derivados | Tem 8 call sites espalhados, alguns críticos (loadFromCloud), outros UX (saveAgenda) | ⚠️ Acoplamento alto, sem barreira de abstração |

**Verdade técnica curta:** a função foi documentada como "sincronizadora" mas opera como "rebuild-com-preservações" — e o set de preservações (done, userEdited, st='paga') é heurístico, não exaustivo.

## 4. O que TIVER que mudar pra v2.10.0

**Opinião técnica direta, sem filtros:**

### 4.1 Obrigatório (sem isso, não é refactor — é cosmético)

1. **Range de meses derivado de `D.cp`**, não de `cMonth/cYear`. A função deve calcular `min(cp.m,cp.y)..max(cp.m,cp.y)` entre cp.st≠'paga'. Isso elimina #1 sem nada mais.
2. **Upsert por chave determinística** (cartão + mês + ano). Substituir o padrão "delete-all + recreate" por: para cada (card,m,y) com cp aberto → cria-se evento e tx **se não existirem**, atualiza-se se existirem (valor, banco, data). Apaga **só** o que não tem cp correspondente.
3. **Dedupe pré-push.** Antes de cada `D.ag.push`, checar se `D.ag.some(e=>e.id===agId)`. Same para `D.tx`. Custo: O(n) por push, irrelevante na escala atual (35 usuários).

### 4.2 Recomendado (qualidade significativa)

4. **Eliminar match por título** ([index.html:3895-3896](index.html:3895)). Após upsert por ID, não há mais necessidade do match em "Fatura " — resolve #2.
5. **Função separada para legacy cleanup.** O cuidado de varrer `venc_*` é uma migração one-shot, não tarefa de sincronização recorrente. Mover para `migrateData` (já há infra para isso, [index.html:1038](index.html:1038)) com flag `_legacyFaturasCleaned=true` no D.
6. **Snapshot e restore mais robusto.** Em vez de `_doneMap` por id, usar `_doneMap` por chave `(card,m,y)` (a mesma do upsert) e migrar legacy `venc_*` para a nova chave. Resolve #3.
7. **`reopenFatura` precisa zerar `agEv.done=false` antes de sincronizar.** Resolve #5.

### 4.3 Opcional (qualidade marginal)

8. **Remover `console.log` da prod** ([index.html:3905](index.html:3905)).
9. **Validação total>0** antes de criar evento (ignorar faturas com estorno zerando).
10. **`save()` após `loadFromCloud`+sincronizar** se houve mudança. Detectar por hash simples antes/depois.

### 4.4 NÃO mudar (atualmente OK)

- Preservação por `done=true` e `userEdited=true` — tese correta.
- Estrutura de IDs determinísticos `agfat_<card>_<m>_<y>` — boa, mantém auditável.
- A função permanecer como uma função única — 90 linhas, lida em uma sentada. **Não fragmentar** em subhelpers só por refactor cosmético.

## 5. Estratégia recomendada

### 5.1 Upsert por chave: SIM, mas com adendos

A estratégia do ROADMAP é correta no espírito, mas **incompleta na execução**. Detalho:

**A chave proposta no ROADMAP_v2.10.0.md (cartão + mês + ano) está certa**, mas:
- **Não é suficiente sozinha.** Precisa vir junto com (a) range derivado de `D.cp`, (b) dedupe pré-push, (c) eliminar match por título.
- A "validação contra dump dos 35 usuários" descrita no ROADMAP item 1 é boa prática — mas o que provar não é "a chave produz N entradas únicas". É: **`D.cp.filter(c=>c.st!=='paga').reduce((acc,c)=>acc.add(c.card+'_'+c.m+'_'+c.y), new Set())` produz exatamente as faturas que o usuário enxerga hoje, sem órfãos ou ausências**.

**Alternativas que considerei e descartei:**

| Alternativa | Por quê descartei |
|---|---|
| **Diff incremental** (calcular diff entre estado atual e desejado, aplicar só o diff) | Excesso de engenharia para 35 usuários. Upsert é mais simples e equivalente. |
| **Cache-aside com TTL** (sincronizar só se >X minutos da última) | Adiciona estado mutável (lastSync), expande superfície de bugs. Não vale. |
| **Event sourcing** (D.ag/D.tx imutáveis, derivados de log de eventos) | Refactor de magnitude muito maior, não cabe em v2.10.0. Talvez no Next.js+Supabase. |
| **Manter "delete-all + recreate" mas com dedupe** | Não resolve #1 (range). Não vale o esforço. |

### 5.2 Shadow mode é fundamental

A proposta do ROADMAP item 2 (shadow mode antes do switch) **deve ser mantida sem flexibilização**. Em 35 usuários reais com 7 ativos por semana, qualquer bug residual atinge alguém em <7 dias. Roteiro:

1. Escrever `sincronizarFaturasEmAbertoV2()` **paralela**.
2. Em todos os 8 call sites, chamar **as duas** funções, mas só a V1 escreve. V2 só calcula e loga diff em `console.warn`.
3. Deploy. Esperar 5-7 dias. Coletar logs (Gileno faria isso manualmente lendo o console; sem telemetria centralizada).
4. Se diff for trivial em todos os 7 ativos, fazer switch (V2 escreve, V1 some).

### 5.3 Canário humano: ordem que faz sentido

Concordo com ROADMAP item 3, com nota: **Gileno → Luan → resto** é a ordem certa. Luan é o usuário que historicamente teve incidentes (sessão de março com transações sumindo); ele é o melhor "barómetro" para detectar regressões reais.

   - **Minha opinião:** separar. Reduz escopo da v2.10.0 e migrações one-shot já têm padrão estabelecido em `migrateData` ([index.html:1038](index.html:1038)).

5. **Telemetria mínima durante shadow mode?**
   - O ROADMAP assume que você lê console manualmente. 7 ativos × 5-7 dias = perda de cobertura.
   - Alternativa low-cost: enviar diffs do shadow mode para um document Firestore `diagnostics/sincronizar_v2/` com TTL de 30 dias. Custo: 1h de implementação + revisão.
   - **Minha opinião:** vale a pena, mas é incremento ao plano. Decisão sua.

## 7. Risco e mitigação

### 7.1 Cenários de corrupção real (não-hipotéticos)

| Cenário | Probabilidade | Impacto | Mitigação atual | Mitigação no refactor |
|---|---|---|---|---|
| **Faturas futuras de parcelamento longo invisíveis** | ALTA (12x parcelado é comum) | Usuário não vê na agenda; pode esquecer de pagar | Nenhuma | Range derivado de cp |
| **Lembrete "Fatura ..." do usuário apagado** | BAIXA-MÉDIA | Perda silenciosa de dado de usuário | Nenhuma (só `userEdited=true` se editado) | Match só por ID determinístico |
| **Duplicata após `reopenFatura`** | BAIXA (operação rara) | UX confuso, dois eventos idênticos | Nenhuma | Dedup pré-push + reset `done=false` no reopen |
| **Done state legacy `venc_*` perdido** | BAIXA (base já migrada) | Evento ressurge como pendente após pago | Nenhuma | Mapping por chave (card,m,y), não por id |
| **Lixo legacy na nuvem após load** | TEÓRICA | Acumula indefinidamente, eventualmente afeta perf | Nenhuma | `save()` após `loadFromCloud` se houve mudança |

### 7.2 Risco do refactor em si

| Risco | Mitigação |
|---|---|
| Refactor V2 introduz bug pior que V1 | **Shadow mode obrigatório** — não negociável |
| Backup do dia anterior não captura estado atual do usuário-vítima | Backup **fresh** minutos antes do deploy (ROADMAP já cobre) |
| Rollback nunca foi testado de verdade | Praticar UMA VEZ antes (ROADMAP já cobre) |
| Mudança de chave migra mal dados antigos | Manter cleanup legacy em `migrateData` separado |
| Concorrência cliente↔nuvem (debounce 2s) cria estados intermediários | **Fora do escopo do refactor.** Bug pré-existente, não-piorado. |
| **Dois clientes do mesmo usuário simultâneos** (desktop+mobile) | **Fora do escopo.** Bug pré-existente. Sem CAS no Firestore, último write vence. Aceitável. |

### 7.3 Risco que NÃO existe (refutando achados antigos)

- **Conversão objeto→array em memória:** **NÃO existe na função atual.** Foi removida em alguma versão entre v2.9.46 e v2.9.53. A função hoje pressupõe `D.tx` e `D.ag` como arrays e cresceria com TypeError se não fossem — mas `migrateData` garante isso na entrada ([index.html:1040-1041](index.html:1040)). Achado histórico do DIARIO está superado.

### 9.2 Onde pode estourar pra 5-6h adicionais

- Se script de validação contra dump local revelar que `D.cp` de algum usuário tem `m`/`y` inconsistentes (ex: `m=13`) — bugs históricos. Conserto = 1-2h por caso.
- Se shadow mode revelar comportamento divergente em base real que não foi previsto em sessão A — volta pra A com mais info. +2-4h.
- Se ao deploy v2.10.0 algum usuário do canário relatar problema → rollback + investigação. +2-3h.

### 9.3 O que vale a pena NÃO fazer

- **Não** reescrever a função em estilo funcional puro (map/reduce/filter chains). O código atual é imperativo e legível. Reescrever em estilo "elegante" introduz risco sem benefício.
- **Não** quebrar em 5 helpers menores. 90 linhas é ok. Quebrar é cosmético e adiciona superfície.
- **Não** adicionar logs estruturados de telemetria além do necessário para shadow mode.
- **Não** mover para módulo externo (`sync.js`). O projeto é arquivo único intencionalmente — manter assim.
